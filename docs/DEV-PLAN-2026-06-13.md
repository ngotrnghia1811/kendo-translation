### 7.1 QA Status Correction — "Not Started" Tag (P0 Priority)

**Problem:** 197,962 segments are marked `qa_approved` from the pipeline import (`metadata.imported_from_pipeline=true`). These were machine-translated, not human-reviewed. From a human workflow perspective, they should show as **"Not started"** (untranslated), not as approved.

**Context:** `reading_progress` table exists but is empty (no reads logged yet). No `article_views` table. The 197,962 `qa_approved` and 19 `translated` segments are real, but only the 19 are human-produced.

**Options:**
- **(a) Display fix only**: Show segments with `metadata.imported_from_pipeline=true` as "Not started" in the UI without changing the DB status. Add a banner in reader/editor: "Machine-translated content, not yet human-reviewed."
- **(b) Status reset**: SQL `UPDATE segments SET status='draft' WHERE metadata->>'imported_from_pipeline'='true'`. This allows translators to claim and edit them. Side-effect: any TM rows seeded from pipeline segments are still `human_approved=false`, so correct.
- **(c) New status value**: Add `pipeline_imported` status. Breaking change to the phase state machine.

**Recommended**: Option (a) for immediate display, option (b) for a proper workflow reset. Option (c) is too costly.

**Effort**: (a) tiny · (b) small + user authorization

### 7.2 Search Enhancement — Preview Window Context

**Problem:** Search results (`/search`) show article title + segment text but no surrounding context. Users must click through to see relevance.

**What to build:**
- In `app/search/page.tsx` (or the results component): below each segment result, show `prev_segment.source_text` + `current.source_text` + `next_segment.source_text` — a 3-segment window.
- The search API (`GET /api/search`) needs to return `position` of each matching segment. Then a second query fetches `position-1` and `position+1`.
- Show as a collapsible "Context ▸" toggle (closed by default) so the list isn't overwhelming.

**Data available:** `segments` table has `position` + `article_id`; segments can be fetched by `article_id` + `position IN [p-1, p, p+1]`.

**Effort**: small (2–3h) — BE endpoint change + FE context expand component.

### 7.3 Document Sorting and Filtering

**Problem:** `/documents` list has no sort or filter controls. All 31 segmented documents appear in insertion order.

**What to build:**
- Sort by: **Name** (title alphabetical), **Length** (segment_count desc/asc), **Progress** (% translated), **Last viewed** (requires data — see below).
- Filter by: **Status** (all / in-progress / complete), **Language** (EN / ZH toggle already in editor).

**View tracking data status:** `reading_progress` table exists but is empty. No `article_views` table. To enable "Most Recently Viewed" sorting, we need to write to `reading_progress` when a reader opens a document. Currently the reader page does NOT write this.

**Recommended approach:**
1. Add sort/filter controls for Name, Length, Progress first (no new data needed — all in existing `articles` table).
2. Add a `POST /api/documents/[id]/view` endpoint that upserts a `reading_progress` row on document open.
3. Wire the reader page to call this endpoint on load.
4. Once data accumulates, expose "Recently Viewed" sort.

**Effort**: (1) small · (2+3) small · (4) deferred.
