### File structure (consistent across all `*_trilingual.md`)

```
# <title> — Trilingual Translation
**Generated:** YYYY-MM-DD HH:MM
**Pages translated:** N/N

---

Page 1

【Heading】
<JA heading>
<EN heading>           ← typically italicized with gloss: *Term* (剣 — sword)
<ZH heading>

---

<JA paragraph>
<EN paragraph>         ← may include [T/N: ...] translator notes
<ZH paragraph>

---

... more paragraphs separated by `---` ...

=== END OF PAGE 1 ===


Page 2
...
```

Key markers:
- `Page N` ... `=== END OF PAGE N ===` — page boundaries
- `---` — paragraph separator within a page
- `【Heading】` — heading marker on the next line(s)
- `[cite_start]...[cite: N]` — inline citation markers (sometimes present)
- `[T/N: ...]` — translator notes
- Each paragraph block has exactly 3 lines: JA → EN → ZH (in that order)

## Schema target (current shape)

From `types/database.ts`:

```ts
Article          { id, title, content_ja, content_en, status,
                   translation_status, quality_score, segmented,
                   segment_count, created_at, updated_at }

Segment          { id, article_id, position, source_text, target_text,
                   source_lang, target_lang, status, locked_by,
                   locked_at, translated_by, reviewed_by,
                   quality_detail, metadata, created_at, updated_at }

DocumentSettings { id, article_id, source_lang, target_lang,
                   paragraph_boundaries: number[],
                   total_segments, translated_count, reviewed_count,
                   approved_count, assigned_translators, created_at,
                   updated_at }
```

DB is **bilingual** (`source_text` / `target_text`), data is **trilingual**.

### Decision needed (backend's call)

1. **Bilingual import (simplest):** drop ZH, store JA→EN. Reader works
   today with no schema change. Lose ~33% of the translation labor.
2. **Bilingual-pair-per-article:** import the same book twice as two
   articles — JA→EN and JA→ZH — sharing `paragraph_boundaries`. No
   schema change. UX-awkward (same book appears twice in listings).
3. **Schema extension:** add `extra_translations jsonb` on `segments` and
   `target_langs text[]` on `document_settings`. Reader stays
   bilingual for now; ZH carried as data for future. Requires migration
   + careful coordination with frontend on `types/database.ts`.

My frontend-lane bias is **Option 1** (ship, iterate). But this is your
domain decision — there may be MAC-RAG / retrieval reasons to prefer 2
or 3.

## Suggested approach (non-prescriptive)

A one-shot importer at `scripts/import-trilingual-references.ts` that:

1. Takes a single `*_trilingual.md` path as arg (start with `baba 1
   clean_trilingual.md` — shorter than `Men Full`).
2. Parses pages → paragraphs → (JA, EN, ZH) triples.
3. Treats each paragraph as one segment with `position` ascending across
   the whole book; emits `paragraph_boundaries` = segment indices where
   a new paragraph starts (every segment in this case, since the
   importer maps 1 paragraph = 1 segment — unless you want to sentence-
   split inside paragraphs, in which case paragraph_boundaries are the
   first-segment indices of each paragraph).
4. Inserts an `Article` (title from filename or H1 line), `Segments`
   (status = `qa_approved` since this is pipeline-finalized content, or
   `translated` if you want to leave QA for humans), and a
   `DocumentSettings` row.
5. Idempotent on re-run (upsert by article title, or skip if already
   imported).

This also serendipitously **solves PARA-BOUNDARIES-IMPL** for this
dataset — the importer writes `paragraph_boundaries`, so the reader's
paragraph-merging logic activates against this content. The separate
PARA-BOUNDARIES-IMPL fix for `app/api/documents/[id]/segmentize/route.ts`
is still wanted for the user-segmentation path.
